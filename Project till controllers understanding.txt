# 1. Why SQL table shows fewer columns but model has more fields?

## Question

Why does SQL Server table show only 5 columns, but model has extra properties like `Office` and `Trips`?

## What it is

SQL table contains only actual database columns:

* `bus_id`
* `office_id`
* `registration_number`
* `capacity`
* `type`

But EF Core also generated:

```csharp id="b8j1y2"
public virtual AgencyOffice Office { get; set; } = null!;
public virtual ICollection<Trip> Trips { get; set; } = new List<Trip>();
```

These are called **Navigation Properties**.

---

## Why we use it / Purpose

They help EF Core understand relationships between tables and allow easy navigation between related data.

Example:

```csharp id="f1v4n8"
bus.Office
```

gets related office data.

```csharp id="w2m8q1"
bus.Trips
```

gets all trips of that bus.

These properties are not physical DB columns.

---

## B. ICollection<T> → Many

# 3. Why `public virtual`?

## Question

Why EF Core generated `virtual` keyword?

## What it does

`virtual` allows EF Core to create proxy classes for:

* Lazy Loading
* Runtime overriding
* Better testing/mocking

---

## Why we use it

Mainly for Lazy Loading.

Example:

```csharp id="h9s2k4"
bus.Trips
```

Trips load automatically when accessed.

Without `virtual`, lazy loading cannot happen.

---

IMPORTANT : If you don't use `virtual`, then **lazy loading** will not work. ✅

But you can still fetch Trips data using:

```csharp id="q9w2m7"
.Include(b => b.Trips)
```

So:

* With `virtual`
  → EF can automatically fetch related data (lazy loading)

* Without `virtual`
  → You must fetch manually using `Include()` or explicit loading.

Yes, if Lazy Loading is properly enabled and `virtual` is used, then usually you do not need `Include()`.

Example:

```csharp id="j2x8v4"
var bus = context.Buses.First();

var trips = bus.Trips;
```

When `bus.Trips` is accessed, EF automatically loads trips from DB.

But this works only if:

* `virtual` is used
* Lazy loading proxies are enabled

Otherwise you must use:

```csharp id="k7m1q5"
.Include(b => b.Trips)
```

exact reply : “We use `virtual` mainly to support Lazy Loading in Entity Framework Core.
It allows EF Core to create a proxy child class at runtime, so when we access related data like `bus.Trips`, EF can automatically fetch that data from the database if it is not already loaded.”

Purpose of the child/proxy class:
It monitors access to virtual navigation properties.
Example:
bus.Trips

When you access this property, the proxy class detects it and automatically loads Trips data from database.

So the proxy class is created to add automatic behavior like Lazy Loading without changing your original class.

# 6. Why `partial class Bus`?

## Question
# What is `partial` in simple words?

`partial` means:

> “One class can be divided into multiple files.”

---

# Why we use it in EF Core?

Because scaffolded files can regenerate.

If you directly edit generated `Bus.cs`, your changes may get deleted when scaffolding runs again.

So EF uses `partial` to let you create another file safely.

---

# Simple Example

### Generated file (`Bus.cs`)

```csharp id="f2m8x4"
public partial class Bus
{
    public int BusId { get; set; }
}
```

### Your custom file (`BusCustom.cs`)

```csharp id="n5q1v7"

{
    public void Print()
    {
    }
}
```

---

# What happens?

C# combines both files into one single `Bus` class automatically.

---

# Real Purpose

It protects your custom code from getting deleted after scaffolding again.

---

# Very short evaluator answer

“We use `partial` so we can extend scaffolded classes in separate files safely without losing custom code during regeneration.”

IMPORTANT:
question : 
For DbSet properties, virtual is mainly used for:

Runtime overriding
Better testing/mocking, how it do runtime overriding tell me with example in simple way

answer:
`virtual` means:
“this method/property can be changed (overridden) by another class at runtime.”

In Entity Framework, EF creates a **proxy class** (child class) from your entity class and overrides virtual properties.

Example:

```csharp
public class Student
{
    public int Id { get; set; }

    public virtual ICollection<Course> Courses { get; set; }
}
```

EF internally creates something like:

```csharp
public class StudentProxy : Student
{
    public override ICollection<Course> Courses
    {
        get
        {
            Console.WriteLine("Load data from database");
            return base.Courses;
        }

        set
        {
            base.Courses = value;
        }
    }
}
```

So when you do:

```csharp
student.Courses
```

EF intercepts it and loads data automatically from DB (lazy loading).

Without `virtual`:

```csharp
public ICollection<Course> Courses { get; set; }
```

EF cannot override it, so proxy class cannot change behavior at runtime.

That is called **runtime overriding**.

Question: but how in dbcontext class it do runtime overriding
answer:
In `DbContext`, EF also creates a **derived/proxy class** during testing or runtime.

Example:

```csharp
public class AppDbContext : DbContext
{
    public virtual DbSet<Student> Students { get; set; }
}
```

Now suppose in testing:

```csharp
public class FakeDbContext : AppDbContext
{
    public override DbSet<Student> Students
    {
        get
        {
            Console.WriteLine("Fake data");
            return base.Students;
        }
        set
        {
            base.Students = value;
        }
    }
}
```

Now:

```csharp
AppDbContext context = new FakeDbContext();

context.Students
```

It will call overridden version from `FakeDbContext`.

That is runtime overriding because:

* parent reference → `AppDbContext`
* actual object at runtime → `FakeDbContext`

So behavior changes at runtime using `virtual` + `override`.

# Why We Shifted from ADO.NET to EF Core

## 1. ADO.NET (Old Way)

In ADO.NET, developers manually wrote:

* SQL queries
* Connections
* Commands
* DataReaders
* Mapping data to objects

Example:

```csharp id="p3x9s8"

    "SELECT * FROM Students", con);

con.Open();

```

## Problems in ADO.NET

* Too much boilerplate code
* Manual SQL writing
* Manual object mapping
* Hard to maintain large projects
* More chances of bugs
* Database-dependent code
* Slower development

---

# 2. Need for Better Approach

Projects became:

* bigger
* enterprise-level
* object-oriented

Developers wanted:

* less SQL
* faster development
* cleaner code
* object-based programming

---

# 3. Entity Framework / EF Core Came

EF Core is an:

# ORM (Object Relational Mapper)

It maps:

* Database tables ↔ C# classes

Now developers work with objects instead of raw SQL.

Example:

```csharp id="d0u9zt"

```

instead of:

```sql id="bl0o9n"

```

# Benefits of EF Core

| ---------------- | ------------------- |

| More code        | Less code           |
| Hard maintenance | Easier maintenance  |
| Low productivity | Faster development  |
| DB-specific      | More DB independent |
